# A/B 测试 —— 为什么本技能"穷尽列举"(并拒绝"模式优先于列表")

这里记录两条输出规则背后的实验,好让这些决定在日后被质疑时站得住。

## 为什么做这个实验
两项技法来自 GSD 的 `codebase-mapper` agent(一种常见的代码梳理思路),在被信任前需要验证:
- **②**"描述**模式**,而不是一份裸列表"(输出纪律)
- **①** 审计员把发现写到磁盘、只返回**计数**(省上下文)

一条听上去有道理、来自成熟项目的原则,不等于它适配*这个*活。所以我们测,而不是断言。

## 设计(一遍,有真值)
- **载体**:一次真实底座-vs-重写(渲染器迁移)的编辑器**控件面**维度 —— 底座 UI 没变,但重写的渲染契约悄悄丢/改了很多控件。
- **真值**:17 个不同的失效/分歧 UI 控件(每个*应该*是一条可动手的发现)。
- **审计员**:冷启动子 agent(无前置上下文),都读两棵树里**同样**的文件。
- **测 ②**:3× **ENUMERATE**("一控件一发现,穷尽,绝不打包")vs 3× **PATTERNS**(② 的措辞)。指标:**召回**(17 个里几个)+ **颗粒度**(发现总数)。*(一次 ENUMERATE 跑因结构化输出重试失败 → 那一臂 n=2;分离仍然清楚。)*
- **测 ①**:固定 12 条发现 → 2× **FULL**(发现内联)vs 2× **COUNTSDOC**(只返回计数,要去读文档)。指标:簇数 + **主导根因**是否一致。

## 结果
**② 模式-不列表 —— 召回中性、颗粒度有害**

| 臂 | 颗粒度(发现数) | 召回(17 个里) | 幻觉 |
|---|---|---|---|
| ENUMERATE | 33, 34 → 均值 **33.5** | **17/17** | 0 |
| PATTERNS | 28, 23, 19 → 均值 **23.3** | **17/17** | 0 |

两臂都*提到*全部 17 个控件(召回保住)。但 PATTERNS 把不同控件**打包**进单条发现(比如把 4 个 cover-scale 滑块折成一条;把字幕模式 + 4 个 FX 控件折成一条),把颗粒度砍掉约 30% —— 而且随着行文越来越"模式化"单调变差(28→23→19)。一次能力对账的交付物是**一处缺口一行可动手**,所以打包是实打实的损失。

**① 计数+读文档 —— 质量中性**

FULL → 5 / 4 簇;COUNTSDOC → 5 / 4 簇;**主导根因一致**("有损的样式契约");评判**中性**。从文档里读回发现,聚类效果和内联一样 —— 信息保住了,所以是个安全的上下文阀门。

## 决定(已进 SKILL.md)
- **拒绝**parity-audit 用"模式优先于列表"。强制**一处缺口一条发现、穷尽列举**。(`codebase-mapper` 的"模式优先于列表"对*描述*代码库对、对*列举*审计错 —— 本技能刻意反着来。)
- **保留**计数+写盘当**默认关的规模化阀门** —— 只有当一次审计塞不进编排者上下文时才用。

## 元教训
那条有害规则来自一个广受好评、被大量使用的工具。名气没让它适配另一个语境 —— 只有 A/B 抓到了它;测前的*断言*(我的)还猜错了失败模式(预测召回更低;真正的伤害是颗粒度更低)。**信一项借来的技法前,先拿它对着真值测。**
