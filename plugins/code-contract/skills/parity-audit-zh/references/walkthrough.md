# parity-audit-zh —— 走一遍(一次实跑)

你有一个已知良好的**底座**和一个**重写/迁移**,需要一份**完整**清单:重写还有哪些没对上 —— 一次性给齐,而不是一条条发现。

## 你做什么(全部用户动作)
```
/parity-audit-zh
```
如果它还不知道两个路径,就告诉它,比如:
> baseline = `<旧系统>/src`(已知良好的原版),new = `<重写>/src`(被审的迁移)

下面全是技能**替你**做的。只有**一处 checkpoint** 它会停下等你签字。

## 一步步发生了什么

**第 1 步 —— 框定(几秒)。** 它判定哪边是底座(判定基准)、哪边是重写,以及"判定基准"—— *哪个可观测产物算"对上"*(渲染器:**导出产物**,不是实时预览)。不用你输入。

**第 2 步 —— 摸清能力地图 → 🟦 你唯一的 CHECKPOINT。** 它读底座,提出 ~8–15 个覆盖整个产品的能力*维度*(如 时序 · 字幕 · 音频 · 调色 · 转场 · 导出 · 编辑器控件面 · …)。它把清单给你。你瞄 ~30 秒:*"漏了哪个轴吗?"* → OK,或加一个。**正是这个小 checkpoint 挡住一整块区域变成永久盲区。**

**第 3 步 —— 审计(并行、自动)。** 一维度一个独立审计员同时跑,各读两套代码库,上报每个*缺失 / 部分 / 分歧*行为,两侧都有 `file:line` + 严重度。这里你啥也不做。

**第 4 步 —— 综合(自动)。** 它把散落的发现收敛成几个**根因**,按严重度排序,把完整清单写到 `docs/audit/parity-<baseline>-vs-<new>-v1.md`,再给你一份摘要。

## 你会拿到什么(示例 —— 数字来自一次真实跑;审计文档本身不随包发布)
> **查了 188 项能力 · 100 项对齐 · 88 处缺口**(5 blocker / 48 major / 34 minor)。88 处收敛成 **6 个根因**;最大那个 —— 一份*有损的样式契约* —— 解释了约一半。外加列出每处缺口(两侧 file:line)的文档,和一份推荐修复顺序(blocker → 主导根因 → 其余)。

## 然后呢
把那份文档直接交给 **`/verified-execution-zh "<文档路径>"`** 去真正关闭缺口,每条都由一个独立验收方确认(见那个技能的走一遍)。

## 如果没有可信底座
直说 —— 审计弱化成自审 / 规格审。parity-audit-zh 的好坏,只取决于它对着衡量的那个参照。
