# 能力对账 —— 简化一次性提示词

一段短、自包含的提示词,不调用完整技能也能触发同一套审计。填上两个路径、(可选)维度,然后粘贴。

---

```
做一次完整的能力对账:BASELINE(已知良好 / 当前在用的参照)vs NEW(重写/迁移)。
底座是判定基准 —— 找出 NEW 没对上它的每一处。不要重新设计、不要"改得更好";只画出分歧。

- BASELINE: <路径 / 分支 / 目录>
- NEW:      <路径 / 分支 / 目录>

先定 判定基准 —— 哪个可观测产物定义了"对上"?(渲染器是 导出产物,不是实时预览;API 是响应契约;UI 是渲染结果。)
对着那个审,而不是手边最方便的表面。

步骤:
1. 速读底座,把产品切成 ~8–15 个覆盖它的正交能力 维度(从底座自己的模块里推 —— 按领域调整)。从四个角度逼出
   覆盖:流水线/入口、数据模型、控件面(每个 UI 控件 / 选项 / 开关 —— 把这些切细列举;一个还显示却不再影响
   输出的控件是最常见的无声重写失地 —— 别把它们埋进一个粗糙的"设置"维度)、输出格式。先列出维度,在昂贵的
   并行展开之前拿到我的快速签字。
2. 对每个维度,并行跑一个独立审计员(一维度一个子 agent)。每个都读两棵树里的真实代码,把它这个维度里每个底座
   能力列举出来,核对 NEW,返回发现:
       { dimension, capability, status: full|partial|missing|divergent, severity: blocker|major|minor|ok,
         baseRef: "file:line", newRef: "file:line" 或 "—", detail }
   硬规则:两侧都标 file:line,否则不算一条发现;别编底座没有的能力;对齐(ok)项也上报,好让覆盖可见。
3. 综合:把散落的发现按 根因 聚类(散落缺口通常共享几个根因 —— 一个层间有损适配器、一个从没接通的配置、一整个
   没实现的模式);浮出主导根因(修一次干掉一片)。按严重度排序。
4. 把完整清单写到一个带版本的文档(docs/audit/parity-<baseline>-vs-<new>-v1.md):每条发现两侧都有 file:line,
   按根因 + 严重度归类。
5. 给一份摘要:头部计数(查了 / 对齐 / 各严重度缺口)→ 根因综合放最前 → 列出 blocker → major 按主题归类(不是平铺)
   → 一份推荐修复顺序(blocker → 主导根因 → 其余),每步对着底座验。

可行的地方,经验地验行为(两边都跑、diff 真实输出),而不是只信读代码。
```

---

## 更短(一行)

> 把 BASELINE `<路径>` 对 NEW `<路径>` 做完整能力对账:切成 ~10 个能力维度,一维度一个并行审计员各读两套代码库,
> 上报每个 缺失/部分/分歧 行为(两侧 file:line + 严重度),然后把缺口按根因归类,给我一份按严重度排序、带修复顺序的清单。
