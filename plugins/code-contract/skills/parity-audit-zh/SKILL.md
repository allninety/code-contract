# 能力对账(Parity Audit)

> 中文版,对应英文版 `parity-audit`。三件套同主线(共享同一 atom-ID):`product-contract-zh`(结构)· `parity-audit-zh`(证据,本技能)· `verified-execution-zh`(闸门)。
> 本文中的"业主"指**产品 owner / 业务负责人 / 非技术决策者**(不是房产业主)。

## 这是干什么的
一次重写 / 迁移 / 移植 / 重构,几乎总会**悄悄丢掉原版做到的东西** —— 一个不再影响输出的控件、一个被不同地夹断的时长、一个变成空操作的导出选项。原版("底座")里往往沉淀着*好几天的调参*,不会白白带过来。最贵的失败模式是**一个一个**地发现这些丢失,拖好几周,每次都像新惊吓("哦,这个也没了?")。这会磨掉信任,让活儿显得没底。

本技能产出相反的东西:**一张穷尽、带证据、按严重度排序、按根因归类的缺口清单**,把全部范围一次性摊到台面上,让剩下的活儿是*被框定的,而不是零碎漏出来的*。它还能把一堆吓人的缺口(比如"88 条")收敛成几个根因(比如"6 个根因;最大那个修一次干掉一半")。

## 核心理念
底座是**判定基准**。任务是找出新实现**没对上它**的每一处 —— 而且靠*两侧都读真实代码*,不靠任何一方的文档或你的记忆。价值按顺序来自三件事:**覆盖**(没有维度漏检)、**证据**(两侧都有 file:line,否则不算一条发现)、**综合**(N 条散落发现 → 几个根因)。

## 本技能是 COMPARE 引擎 —— 冻结规格用 `product-contract-zh`
parity-audit-zh 是一对里的**深度 / 证据**那一半。它的活儿是把一个实现对着参照做 COMPARE,产出按严重度、带 file:line 的缺口清单。那份*持久的规格* —— 把一个产品的能力冻结成有命名、业主↔代码可追溯的契约 —— 是 **`product-contract-zh`** 的活儿(结构那一半)。两者共享同一套稳定 atom-ID、互相组合;别在这里重建契约。

这次审计有两种起法:
- **已有契约 → 对着它审(首选)。** 读 `product-contract-zh` 的 atom 登记表,**逐 atom** 审 —— atom 就是你的清单,它们的 trace link 把你指向 target 的代码。**别**自己另派一套维度;再推导的漂移正是两份审计在颗粒度上吵架的根因。把每个 atom 在该 target 的状态 + 证据写到 `audit-results/ATOM-STATUS-<target>.md`(用 atom-ID 作键 —— 绝不写回契约词典的 `状态` 列,那是 product-contract-zh 的单产品快照)。若发现一个**没有任何 atom 代表**的真实承诺,把它记为下一轮 `product-contract-zh` 的契约缺口 —— 别私自扩大审计范围。
- **没有契约 → 自行建立能力地图(冷启动),事后可顺手产出一份。** 按下面的方法做发现、建维度 / 清单。若之后想要持久规格,把审过的能力(不只缺口)导成**草稿 atom** 交给 `product-contract-zh` 归一化。这也是"动工前先框定迁移范围"的用法。

## 什么时候跑 + 唯一前提
- **迁移的中后期**(主模式)—— 新实现已经*尝试*过大部分功能,这样缺口是真分歧,而不是"还没做"。
- **早期、先框范围** —— 只跑第 1 阶段(清点 + 维度),产出就是重写必须命中的清单。若要*持久*的前期规格(有命名、可追溯、跨次复用),改用 `product-contract-zh` 去 FREEZE,再对着它审。
- **周期性、当回归闸** —— 大改动后重跑,抓新引入的漂移。

唯一前提:**必须存在一个可信、可指认的底座。** 本技能的好坏取决于它的参照。若"底座"本身有 bug / 不完整,或你确实分不清哪边权威,就直说 —— 审计退化成自审 / 规格检查,仍有用但发现更弱。别把一个站不住的参照冒充权威基准。

## 方法

### 1. 框定审计 —— 并定义判定基准
开任何 agent 之前先钉死:
- **底座**:参照(路径 / 目录 / 分支 / commit / tag,或一份规格)。"正确"是什么意思。
- **新实现**:被审的重写 / 迁移。
- **判定基准** —— *哪个可观测产物定义了"对上"?* 这比听上去重要。对渲染器是**导出产物**,不是实时预览(用户拿到的是导出;底座常有预览和导出两条路,二者必须一致,而真正交付的是导出)。对 API 是响应契约;对 UI 是渲染结果 + 行为。对着**那个产物**审,而不是手边最方便的表面 —— 否则一个冷启动审计员会去比预览,漏掉只在导出里的分歧。

明说:底座是标准;目标是找分歧,**不是**重新设计。改进留到对齐之后、有意识地做(见原则)。

### 2. 摸清能力地图 —— 然后在展开前 CHECKPOINT
这是成败一步。(一次*非正式*冷启动里 —— 无人工上下文 —— 当覆盖被主动逼出来时,框架 + 维度迁移得不错;唯一的失手是把控件面切得不够细。这没留正式记录,所以当观察看,不当实测结论 —— 不像 enumerate-vs-patterns 的 A/B 那样有据。)分两步:

**(a) 清点 → 维度。** 读底座源码,建一份能力清单,折成 ~8–15 个**正交维度**,以小重叠覆盖整个产品。从底座**自己的模块**里推 —— 绝不用通用清单。为了*逼出*覆盖,从四个角度走一遍底座并确认每个都被代表:
- **入口 / 流水线** —— 从头到尾会跑什么;
- **数据模型** —— 用户能设的每个字段都是一个必须被渲染的行为;
- **控件面** —— 每个 UI 控件 / 配置项 / 开关(见下面的原则 —— 这是重写悄悄阵亡的地方,也是你一旦把它揉进一个粗糙的"设置"维度就最容易少算的轴);
- **输出格式** —— 每一种产出物(文件类型、分辨率、导出、字幕、旁路通道)。
一个你忘掉的维度,就是一个永久盲区 —— 审计永远不会往那看。

**(b) Checkpoint —— 停下来等(别跳过)。** 把维度清单(连同每个覆盖的底座文件)摆出来,**在启动任何审计员之前拿到人的签字**。在他们回话前别调展开工具 —— 在这里加 / 删一个轴,正是用户控制跑多少 agent、覆盖什么的手段。这一步*容易被跳*(agent 一启清单就再没人看)—— 顶住;跳过它是一次无声失败。便宜的保险:懂这产品的人几秒就能看出漏了哪个轴。若真无人值守,也要把清单 + 你自己的覆盖疑点发出来,好让"跳过"事后可见。

维度示例(按领域调整):
- *媒体 / 渲染应用*:时长与时序 · 转场 · 文字 / 字幕 · 音频 · 调色 · 布局变体 · 比例与分辨率 · 导出 · 编辑器控件。
- *Web API*:端点 · 鉴权 · 请求校验 · 错误形状 · 分页 · 限流 · webhook · 幂等。
- *数据流水线*:摄入 · 变换 · schema/校验 · 去重 · 调度 · 重试 · 输出 · 可观测性。
- *UI / 组件库*:每个组件的 props · 状态 · 无障碍 · 主题 · 事件 · 响应式。

### 3. 一维度一审计员,并行展开
每个维度配自己的审计员 agent。它们互相独立,并发跑,整张地图很快。给每个审计员:
- 底座路径 + 新实现路径,以及它这个维度要比的**具体文件 / 符号**;
- 指令:判之前在**两棵树里都读真实代码**;
- 发现 schema(见下)和分类法;
- 硬规则:**两侧都标 file:line**,且**别编**底座没有的能力;
- 输出纪律:**一处缺口一条发现 —— 穷尽列举;绝不把几个控件 / 行为打包成一条"模式"发现。** (本技能的一次 A/B 表明打包会保住*召回*,但把单项*颗粒度*砍掉约 30% —— 比如把 4 个 cover-scale 滑块折成一行 —— 而缺口清单的存在就是为了逐行去动手,所以这种损失是真的。)每条发现只写**当前状态**(不写"曾经 / 过去是"),并**给出处方**(说新实现要做到什么才算对上)。*注:"模式优先于列表"这条常见代码梳理启发法(如 GSD 的 `codebase-mapper`)对*描述*代码库是对的,对*列举*审计是错的,所以本技能刻意反着来(由一次 A/B 验证,`references/ab-test-enumerate-vs-patterns.md`)。*

**规模化阀门 —— 省上下文(默认关)。** 默认:审计员把完整发现返回给编排者 —— 它需要内联才好聚类。只有当一次审计大到塞不进编排者上下文时,才切到:每个审计员把发现**写**到 `docs/audit/<dimension>.md`,只返回计数 + blocker/major 项;第 4 步综合再去**读**那些分维度文档。一次 A/B 确认这样不损综合质量,所以是个*安全*阀门 —— 但它多了一次写→读,所以非到上下文真扛不住别用。

实现优先级:
- **有 Workflow 工具** → 在一个 `parallel([...])` 里一维度一个 `agent(...)`,各带结构化输出 schema。(骨架:`references/workflow-skeleton.js`。)最干净。
- **有子 agent(Agent 工具)** → 在同一条消息里把各维度审计员一起启,好并发。
- **都没有** → 维度顺序做。慢些,同样严谨。

### 4. 综合(价值最高的一步)
收齐所有发现,然后:
- **根因聚类** —— 散落的缺口通常共享几个根因(两层之间一个有损的适配器 / 契约;一个从没接通的配置值;一整个没实现的模式)。按根因归类。这才是把"88 条"变成"6 修"、并浮出**主导**根因的办法 —— 修一次干掉一片。先做这个,再展示任何东西。
- **严重度排序** —— `blocker`(破交付 / 出错输出)> `major`(可见能力缺口)> `minor`(外观 / 边角)。
- **落一份带版本的文档** —— 把完整清单(每条发现、两侧 file:line)按根因 + 严重度写到一个带日期 / 版本的文件(如 `docs/audit/parity-<baseline>-vs-<new>-v1.md`)。一条不丢;范围有一个可引用的落点。
- **给一份摘要** —— 在对话里给:头部计数(查了 / 对齐 / 各严重度缺口);**根因综合**放最前(最有用);blocker 逐条列出;major **按主题归类**(不是 50 条平铺);以及一份**推荐修复顺序**(blocker → 主导根因 → 其余),每步对着底座验。附上完整文档链接。

## 发现 schema
每个审计员返回这样形状的条目(harness 支持就用结构化输出):
```
{
  "dimension":  "<被审的轴>",
  "capability": "<一个具体的底座行为/功能>",
  "status":     "full | partial | missing | divergent",
  "severity":   "blocker | major | minor | ok",
  "baseRef":    "<底座里的 file:line>",
  "newRef":     "<新实现里的 file:line,缺失则 '—'>",
  "detail":     "<具体差异:底座做什么 vs 新实现做 / 缺什么>"
}
```

## 状态与严重度分类
- **status** —— `full`:新版与底座行为一致。`partial`:有但不完整。`missing`:底座有、新版没有。`divergent`:有但行为/算法不同(永远写清差异)。
- **severity** —— `blocker`:挡交付或产出错误输出(如一个静默无效的设置、一条会损 / 截输出的路径)。`major`:真实、用户可见的能力缺口。`minor`:外观或边角。`ok`:已对齐 —— **照样上报**,好让覆盖可见(读者看到*做好了什么*,不只坏了什么)。

## 业主↔代码可追溯在 `product-contract-zh` 里
一个能力有两张脸 —— *应用*脸(业主触达的)和*代码*脸(实现它的),用一个稳定 ID 连起来,好让业主和 agent 指着同一个东西 —— 而最伤的缺口正活在两者*之间*(业主看得见的控件,却没有代码读它)。那份持久、有命名、双语的记录(功能词典、业主脸大白话纪律、默认可用 / 遇阻才改)是 **`product-contract-zh`** 的活儿,不是本技能的。有契约时,本审计只填每个 atom 在该 target 的**状态 + 证据**到它自己的 `audit-results/ATOM-STATUS-<target>.md`(用 atom-ID 作键)—— 词典的 `状态` 仍是 product-contract-zh 的单产品快照,所以每个 target 各有自己的状态文件、互不双写。没契约时,你的发现可以引导出一份。无论哪种,保持 ID 稳定,好让业主把任何发现倒查回他认得的功能。

## 原则(为什么)
- **底座是标准 —— 忠实移植,别重新发明。** 重写最常见的失地方式,就是作者重造核心逻辑、丢掉原版烤进去的调参。审计时*描述*分歧;别就地"改得更好"。一个你认为是改进的分歧也是一条发现 —— 标出来,事后有意识地决定。
- **控件面是重写悄悄阵亡的地方。** 一次重写最常见的*隐形*失地,是一个 UI 里还在、却不再影响输出的控件 —— 一个啥也不干的滑块、一个空操作的设置、一个被新契约悄悄剥掉的选项(常因一个改名/丢字段的有损适配器)。把控件逐个列出,确认每个真的驱动输出;别把它们揉进一个粗糙的"设置 / 编辑"维度。实践中这常是最大的缺口簇,也是维度一粗就最容易少算的。
- **两侧都有证据,否则不算发现。** 底座 + 新实现都标 file:line(缺失则 `—`)。这才让清单可信、能立刻动手,也挡住"感觉不一样"的空话。
- **报对齐,不只报缺口。** 列出 `ok` 项能显示覆盖和比例 —— 既安心又校准。
- **综合胜过列举。** 88 条平铺让人泄气、没法动手;同样 88 条归成 6 个根因就是一份计划。永远聚类。
- **经验地、对抗地验行为。** 读代码会误判运行时;"我实现了 X"不是 X 能用的证据 —— claim 记的是你*说*做了什么,不是代码做了什么,两者常不一致。可行时就两边都跑、diff 真实输出(一帧、一个响应、一个文件)。验一个缺口*是否关闭*时,取目标倒推姿态:在新实现真实输出对上底座之前,默认它没修好。
- **别编底座没有的能力。** 每条发现都扎在底座代码里。新实现*多*出来的东西没问题、不算缺口。

## 坑(已知失败模式 + 解法)
- **把别处的原则过度移植。** 一条在别处对的规则在这里可能错。*真实案例:*"模式优先于列表"对*描述*代码库对、对*列举*审计有害 —— 一次 A/B 抓到它把单项颗粒度砍约 30%(`references/ab-test-enumerate-vs-patterns.md`)。**解法:**信一项借来的技法前,先拿它对着真值测;来源的名气 ≠ 适配你的活。
- **维度太粗,藏住最大的簇。** 把控件面折进一个"设置"轴,会少算最常见的重写失地 —— 还显示却不再驱动输出的控件。**解法:**把控件面切细(第 2 步),展开前 checkpoint 维度。
- **信读代码胜过信行为。** "看着接上了"≠"能用"。**解法:**至少对 blocker,两边都跑、diff 真实输出。
- **没有可信底座。** 参照本身站不住,每条发现都继承那份疑。**解法:**直说;退化成显式的自审 / 规格审,而不是冒充权威基准。
- **靠计数宣布审计完成。** 计数不是覆盖,声称不是证据。**解法:**做下面的验收步。

## 验收 —— 在有人据它行动前先确认审计可信
一份会被据以行动的审计,本身必须可信;"我产出了一张清单"不是清单正确的证据。交付或行动前:
- **抽查证据** —— 随机抽一批发现,确认所引 `file:line` 存在、且两侧都如发现所说。没有可核证据的发现是噪声。
- **常识校验覆盖** —— 对齐数合理吗?每个维度真的跑了吗(没有静默失败的 agent)?一个返回零发现的维度可疑 —— 去确认,别假定对齐。
- **压测主导根因** —— 它真能解释分给它的那些发现吗?若把它修了,那些发现真会关闭吗?
- **经验地验 blocker** —— 对每个 blocker,两边都跑、diff 真实产物(一帧、一个响应、一个文件)。别交一份你只是读出来的 blocker 清单。

## 输出模板(落盘文档)
```
# 能力对账:<baseline> vs <new> — v1
> 生成:并行审了 <N> 个维度 · 查了 <X> 项能力 · <Y> 项对齐 · <Z> 处缺口
> (blocker B / major M / minor m)。底座 = 交付标准;本文只列 <new> 分歧之处。

## 根因综合(Z 处缺口收敛到约 K 个根因)
1. <主导根因> — <它解释了哪些发现,估计数> — <一句修复方向>
2. ...

## BLOCKER (B)
### 1. <能力>  [status]
- 底座:`file:line`  ·  新版:`file:line`
- <detail>
...
## MAJOR (M)  — 按根因/主题归类
## MINOR (m)
```

## 交付时的语气
若这次审计是因为重写"感觉不如老的"才触发,诚实的完整计数可能刺痛 —— 但一张完整的地图*印证*了那种感觉、也显示你当真,远胜于零碎漏缺口。把坏消息和根因综合、修复顺序一起给,它读起来就是一条路线,而不是一纸判决。
