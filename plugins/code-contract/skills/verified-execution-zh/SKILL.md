# 验收式执行(一个强制闸门)

> 中文版,对应英文版 `verified-execution`。三件套同主线(共享同一 atom-ID):`product-contract-zh`(结构)· `parity-audit-zh`(证据)· `verified-execution-zh`(闸门,本技能)。
> 本文中的"业主"指**产品 owner / 业务负责人 / 非技术决策者**(不是房产业主)。

## 这是干什么的
要消灭的失败模式:实施者(人或 AI)说"100 做完了",但实际只做了 40,打了折,中途跑去修一个小 bug,最后落在 15 —— **而且没人看得见**,
因为那份计划 / 清单由同一个实施者掌握,他可以悄悄改掉或放弃。口头承诺、一份 TODO、一段记忆、"我会照计划做" —— 全是**软的**:
实施者控制它们,所以它们根本不是约束。

本闸门让"完成"= **一个独立验收方,读真实的代码 / 产物,对照一条书面、可观测的验收标准确认过 —— 而且结论进了一本不由实施者单方掌控、改动会留痕的账本。**

## 唯一让它"有牙齿"的原则
**实施者不掌控这道闸门。** 牙齿只来自三处,其余都是表演:
1. **一个独立验收方** —— 一个*独立、全新上下文*的 agent,带*对抗性*提示词,去读**真实**的代码 / 产物,而不是实施者的总结。
2. **确定性、改动会留痕的管道** —— 契约和通过 / 失败的账本是**文件 + 脚本**,不是某个 agent 的一句话。结论流向:验收方 → 脚本 → 账本。(`scripts/harness.py`)
3. **人是最终闸门** —— 你读账本,没有你,谁都不能发布。闸门的职责是让实施者的真实状态**无法对你隐藏**。

推论(要守住):**判断交给 agent,不容糊弄的部分交给脚本。** 多加 agent 就是多加交接、多几处跑偏。把 agent 数量压到最少,把改动会留痕的部分塞进脚本。

## 最少角色(别再加)
| 角色 | agent 还是机制 | 做什么 |
|---|---|---|
| **实施者 / worker**(通常就是主会话) | agent | 一次一条,最小改动,写一条 CLAIM。**不许给自己标完成。** |
| **验收方** | agent —— *独立、全新上下文、对抗性* | 读契约条目 + 原始 diff + **跑** DoD;返回结构化 PASS/FAIL + 证据。**牙齿在这里。** |
| **契约 + 账本 + 闸门** | **机制**(`scripts/harness.py`,追加写文件) | 确定性地记录 claim/verdict;算出闸门状态;结论无可争辩。 |
| **你** | 人 | 最终闸门 —— 读 `status`,可以否决任何一条。 |

仅在规模化时加(几条不用;条目很多且相互影响时再加):一个**集成审计员** —— 周期性做系统级巡检,看关闭 A 有没有弄坏 B、账本和现实是否一致。否则别加第 5、第 6 个角色。

## 契约与验收标准(DoD)
每条都需要一个**可观测**的 DoD —— 一个你能**跑**或能**看**的测试,且绑定它要对齐的底座 / 规格 —— 不是"代码看着对"。读代码会误判行为(假"完成"就是这么来的)。好 DoD 的例子:
"设 anim=Blur,渲染第 N 帧;字幕带模糊入场(和 anim=fade 不同)";"用坏 body 发 POST /x,返回 422 且 `{error}` 与底座一致";"跑测试 T,它通过"。契约是**只追加**的 —— 事后改某条的 DoD 就是挪门柱,所以脚本对 DoD 做哈希:验收时不一致会标记;**而且如果某条在 PASS 之后 DoD 被改了,`status`/`gate` 会把它从 VERIFIED 降级为 STALE** —— 挪过的门柱藏不到一个旧的 ✓ 后面。

## DoD 是"业主↔代码"的桥
好的 DoD 用**业主可观测**的话来写(你会看到或会跑什么),但对着**代码**校验 —— 所以它是*应用*维度(业主)和*代码*维度(agent)之间的共同语言。当条目来自一份 **`product-contract-zh` atom 登记表**(共享主线),或一份对齐它的 `parity-audit-zh` 缺口清单时,**复用那套稳定 atom-ID** 当契约条目 ID。于是*冻结 → 审计 → 关闭 → 验收*全围着同一个 ID 转:product-contract-zh 冻结了这个 atom、parity-audit-zh 在这个 target 上发现它没通过、本闸门把它关闭。一次 PASS 只对**指定 target + DoD** 关闭这个 atom —— 不删 atom、也不抹掉它在其它 target 的审计证据。业主能把任何一条结论倒查回他认得的功能,也能把任何功能查到它的结论。(主线协议:`product-contract-zh` 的 `references/three-skill-coordination.md`。)

## 循环(怎么用)
```
export HARNESS_DIR=docs/product-contract-v1/execution-ledger     # 把账本和契约包放一起
harness.py init                                  # 建账本目录(contract.jsonl, ledger.jsonl);默认 ./.harness/
#   (HARNESS_DIR 对下面每条命令生效;单条命令也可用 `--dir <path>`)
harness.py import items.jsonl                     # 批量加条目(每行一个 {id,title,ref,dod})
#   (或一条条加:)  harness.py add G1 "标题" "底座 ref" "DoD"
# 对每一条:
harness.py claim G1 "我改了什么(file:line)" --by worker   # worker 记录意图;不能给自己判过
#   → 做最小改动
#   → 启一个验收方子 agent(全新上下文),给它:契约条目 + 原始 diff + 下面那段对抗性提示词
harness.py verdict G1 PASS "证据:渲染出模糊≠fade,和底座一致" \
           --dod "<同一段 DoD 文本>" --by verifier   # 记录验收方结论;对 DoD 哈希做防篡改
harness.py status --write                        # 人读这个;--write 还会把 status.md 落到账本目录
harness.py gate                                  # 只要有一条不是 PASS 就非零退出 —— 实打实地拦住
```
- **FAIL → 这条重新打开,worker 重做。PASS → 完成。** worker 从不亲手敲自己的 PASS —— 结论是验收方的,由脚本记录。
- **岔路进 backlog,别就地处理。** 干到一半发现别的 bug → `harness.py add` 进去(或 spawn_task)—— **别**丢下当前这条去追它。追岔路正是把 100 变成 15 的那一步。
- **自证守卫:** `claim --by` 和 `verdict --by` 若是同一身份,会标 `⚠SELF-CERT`(自己验自己没有牙齿,去找独立验收方)。

## 验收方提示词(粘进验收方子 agent —— 牙齿在这里)
> 你是一个**没有任何前置上下文**的独立验收方。**在真实代码 / 产物证明之前,默认这条没做完。** 给你的是:契约条目(做什么 + 底座/规格 ref + DoD)和 worker 改动的原始 diff/文件。
> 别信 worker 的话 —— claim 记录的是他*说*了什么,不是代码实际做了什么。自己去读底座/规格 ref 和新代码,**跑或观察 DoD**,再判。只有你独立确认 DoD 成立才返回 `PASS`;否则 `FAIL` 并给出还差在哪的具体证据。结构化输出:{verdict, evidence, dod}。

## 有牙齿还是表演?(用来评判任何闸门 —— 包括本技能)
任何一条"否"都意味着它是表演、没牙齿:
- 验收方是不是一个*没做过这活*的*不同* agent(全新上下文)?
- 它读的是*真实代码 / 产物*,还是实施者的总结?
- "完成"是一个*可观测、可运行*的测试,还是"看着对"?
- 契约 / DoD 防得住*悄悄*改吗(只追加 + 哈希校验)?
- 闸门是真的*拦*(FAIL 就停),还是只是提醒一下?
- *你*能看到全部状态并否决它吗?

## 坑(已知失败模式 + 解法)
- **验收方读的是 claim,不是现实。** 那就成了橡皮图章。给它原始 diff + DoD,绝不给我的叙述。
- **DoD 是"看着对"。** 不可验证 → 假 PASS。每条 DoD 都要可运行 / 可观测。
- **悄悄改契约。** 把 DoD 挪到"已做"的样子,就把牙齿拔了。只追加 + 哈希标记 + STALE 降级。
- **闸门只是建议。** 不拦的闸门只是提醒。用 `gate` 的退出码;人来执行。
- **跳过人。** agent 让状态*可见,好让你否决* —— 它们不替代你。永远别把最终闸门交出去。
- **角色膨胀。** agent 越多越跑偏。两个 agent(worker、验收方)+ 脚本 + 你,就是全部。

## 验收(对闸门本身)
- 确认验收方真的**跑了** DoD —— 它的结论里必须有具体观测到的证据,不是"看着没问题"。
- 抽查一条 PASS,自己重跑它的 DoD;账本必须和现实一致。
- 确认没有哪条的 DoD 在加入后被改过(`harness.py` 会标 STALE)。

## 诚实的边界
单个 Claude 会话里,编排者仍然是 Claude(它启验收方、跑脚本)。所以这**不是**硬件级约束。它真正的牙齿是:(1) 验收方的*独立性*(一个全新、对抗性、读现实的 agent,比自检难骗得多);(2) *确定性*的契约/账本/闸门(脚本的结论吵不动,而且改动可见 —— 两个经典的赖账动作会被自动暴露:**给自己判** 会被 `verdict --by` 对上 worker 时标出来,**PASS 后挪门柱** 会把条目降级为 STALE);(3) *你*是最终闸门。它让假"完成"难得多、且永远*可见* —— 不是不可能。这份可见,加上你,就是全部意义。

## 脚本与驱动(免得你手动跑 N 遍)
`scripts/harness.py` —— 纯标准库 Python,只追加写 `{contract,ledger}.jsonl`(默认 `./.harness/`;用 `--dir <path>` 或 `$HARNESS_DIR` 改到契约包的 `execution-ledger/`)。子命令:`init`、`add`、`import`、`claim`、`verdict`、`status`(`--write` 还会落 `status.md`)、`gate`。`--help` 看用法。

条目很多时(比如 88 条缺口),别手动跑循环。用**驱动** —— `references/driver.js`,一个 Workflow 骨架,自动对每条跑 `worker → 独立验收方 → harness verdict`(流水线;worker 在隔离 worktree 里跑,免得并行改动打架)。驱动负责编排、agent 调 `harness.py`;**你的精力停在批次层**:把 DoD 清单一次性过一遍,然后读 `status`、抽查样本、用 `gate`。你是最终闸门,不是循环操作工。
