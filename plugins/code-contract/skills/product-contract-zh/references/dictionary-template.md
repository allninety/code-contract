# 功能词典 —— 模板 + 规则

词典是业主↔代码的桥:每行一个 atom,两张脸用一个稳定 ID 连起来。列做了分组,**业主块连续靠左**(ID + 中文/English/语义/在哪 + Feature Atom),**agent 块靠右**(代码锚点 + Code Atom),`状态` 是收尾的单产品快照。

> 下面的示例行来自一个渲染 / 导出类应用 —— **仅作示例**,按你的领域改写。**双语两列(中文名 / English)按业主语言取舍**:业主跨两种语言才两列都留;单语项目去掉其一。

## 表
```text
|       ┌──────────────── 业主侧 ─────────────────┐        ┌── agent 侧 ──┐  快照
| ID    | 中文名      | English          | 语义(大白话:它管什么/你会看到什么)        | 在哪(你触达)       | Feature Atom      | 代码锚点(file:line)        | Code Atom         | 状态     |
|-------|------------|------------------|------------------------------------------|-------------------|-------------------|---------------------------|-------------------|----------|
| F-res | 导出分辨率  | Export resolution | 出片的清晰度:1080/2K/4K,挑哪个出片就多大  | 导出面板·分辨率下拉 | EXPORT.resolution | Film.tsx:36 · render.cjs:54 | CODE.exportPipe   | 🔴 回归  |
| F-fx  | 特效字幕    | FX caption       | 字幕用关键词大字+特效入场,而不是整句平铺    | 样式·字幕模式       | CAPTION.fxKeyword | buildCaption.ts:101        | CODE.captionBuild | 🟡 待验证 |
```

## 规则
- **短、能撑过讨论的 ID**:`F-res`、`F-floor`、`F-fx`、`F-srt-source`。这个 ID 是贯穿词典、atom 登记表、parity 状态、verified-execution-zh 结论的连接键 —— 永不重新编号。
- **语义 / 在哪 / 名字都用大白话**,绝不用术语。检验:一个从没看过代码的人能看懂吗?`style.anim` 不合格;"字幕怎么出现"合格。
- **代码锚点是 agent 的脸** —— `file:line`,可列多个。这是唯一允许技术化的列。
- **`状态` 是单产品快照、单写。** 它回答"*这个*产品当前是否兑现了承诺"—— 不是按 target 的字段。当你审 A/B 多个 target 时,**parity-audit-zh** 把各 target 的状态写到 `audit-results/ATOM-STATUS-<target>.md`(同一 atom-ID),绝不写回这个 `状态` 列。
- **DoD 不在这。** 一行的可观测验收测试(DoD 种子)活在 `USER-FEATURE-ATOMS.md` 里(面向业主、大白话),不在词典 —— 见 `traceability-template.md`。
- **一行 = 一个 atom。** 一行要用"和"才能描述两个承诺,就拆成两行。
- **默认可用、遇阻才改。** agent 把每一行写成可用并立即上线;业主只标看不懂的行,agent 去修那些。**不逐行签字。** 名字一旦定了就复用 —— 绝不给同一个东西起第二个名字。

## 状态标签(少而一致)
- `✅ 保留` —— 在且可追溯。
- `🟡 待验证` —— 已映射,但还需要 fixture / 运行时证据 / 人工检查。
- `🟠 未接线` —— 模型/UI 在,但消费链不完整。
- `🔴 回归` —— 产品承诺没兑现。
- `🟣 新增` —— 底座之外的新能力;需独立验收。
